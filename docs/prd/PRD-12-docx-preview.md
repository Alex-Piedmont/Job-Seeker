# PRD: WYSIWYG Resume Preview via docx-preview

**Project:** Job Seeker
**Date:** 2026-03-04
**Status:** Draft

---

## 1. Brief Overview

The resume preview currently renders markdown with `ReactMarkdown` and Tailwind's `prose prose-sm` classes in two locations: the wizard step-review panel (`src/components/resume/wizard/step-review.tsx`, 307 lines) and the application drawer's resume editor preview mode (`src/components/resume/resume-editor.tsx`, 69 lines). The downloaded `.docx` is generated by `src/lib/docx-generator.ts` (282 lines) using the `docx` npm package, which applies Calibri font, navy/blue color scheme, uppercase section headers with bottom borders, centered contact info, and custom spacing — none of which appear in the in-browser preview.

This feature replaces the `ReactMarkdown` preview with an in-browser rendering of the actual `.docx` binary using the `docx-preview` library. The user sees exactly what they will download: same fonts, colors, spacing, and layout. The markdown edit mode is preserved unchanged.

---

## 2. Key Requirements

- **WYSIWYG preview:** The in-browser resume preview shall render the actual `.docx` output, not a markdown approximation. What the user sees shall match what they download.
- **Shared generation path:** The preview shall use the same `markdownToDocx()` pipeline as the download, ensuring zero visual drift between preview and download.
- **Preview in both contexts:** The docx preview shall replace `ReactMarkdown` in both the wizard step-review panel and the application drawer's resume editor preview mode.
- **Edit mode preserved:** The markdown `Textarea` edit mode in `ResumeEditor` shall remain unchanged. Only the "Preview" tab switches to docx rendering.
- **Loading state:** A loading indicator shall appear while the `.docx` blob is being generated and rendered, since `docx-preview`'s `renderAsync` is asynchronous.
- **Dark mode handling:** The docx preview container shall use a white background regardless of the app's dark mode setting, since the `.docx` output is always black text on white.

---

## 3. Acceptance Criteria

### Core preview

- [ ] A new `DocxPreview` component exists at `src/components/resume/docx-preview.tsx` that accepts `markdown: string` and `className?: string`, generates a `.docx` blob client-side, and renders it into a container div using `docx-preview`'s `renderAsync()`.
- [ ] `docx-generator.ts` is refactored to export `buildDocument(markdown): Document` (shared). `markdownToDocx()` remains unchanged, calling `buildDocument()` then `Packer.toBuffer()`.
- [ ] A new `src/lib/docx-generator.client.ts` imports `buildDocument()` and exports `markdownToDocxBlob(markdown): Promise<Blob>` using `Packer.toBlob()`.
- [ ] The rendered preview displays Calibri font, navy name, uppercase section headers with bottom borders, blue/black experience headers, italic meta lines, and bullet points — matching the downloaded `.docx`.
- [ ] A spinner is shown while `markdownToDocxBlob()` and `renderAsync()` complete. Initial render has no debounce delay; subsequent re-renders debounce at 300ms.
- [ ] The preview re-renders when the markdown content changes (e.g., after user edits in edit mode and switches back to preview). The container is cleared (`innerHTML = ""`) before each re-render.
- [ ] Empty or whitespace-only markdown shows a placeholder ("No content to preview") instead of an empty white box.
- [ ] If docx rendering fails, the component falls back to `ReactMarkdown` rendering (graceful degradation).
- [ ] A `styleElement` ref is passed to `renderAsync()` to scope `docx-preview` styles per instance, preventing conflicts when multiple instances are mounted.

### Integration — Wizard Step-Review

- [ ] `step-review.tsx` imports `DocxPreview` via `next/dynamic` with a skeleton loading placeholder.
- [ ] The `<div className="prose ..."><ReactMarkdown>` block (lines 166-169) is replaced with `<DocxPreview markdown={generation.markdownOutput} />`.
- [ ] The left/right layout (resume left, scorecard right) and `max-h-[70vh]` scroll behavior are preserved. Height is controlled via `className` prop.

### Integration — Resume Editor

- [ ] `resume-editor.tsx` imports `DocxPreview` via `next/dynamic` with a skeleton loading placeholder.
- [ ] The `ReactMarkdown` preview branch (lines 63-65) is replaced with `<DocxPreview markdown={editedMarkdown} />`.
- [ ] The edit/preview toggle buttons and reset button continue to function identically.
- [ ] Switching from edit to preview re-renders the docx preview with the latest `editedMarkdown`.

### Dark mode

- [ ] The docx preview container has a white background (`bg-white`) and is not affected by `dark:prose-invert` or other dark mode styles.
- [ ] The container has a subtle border or shadow so it reads as a "document" embedded in the UI, in both light and dark themes.

### Performance

- [ ] The docx generation + render cycle completes within 1 second for a typical 1-page resume on a modern browser.
- [ ] The preview debounces re-renders at 300ms after markdown changes. Initial mount renders immediately (no debounce).

### Preservation

- [ ] The download button and download flow are completely unchanged.
- [ ] The markdown edit mode (`Textarea`) is unchanged.
- [ ] `ReactMarkdown` import can be removed from both files if no longer used, keeping the bundle clean.
- [ ] All existing vitest tests pass.
- [ ] `docx-generator.ts` continues to work in the server-side download route without regression.

## 4. Out of Scope

- PDF generation or PDF preview. The download format remains `.docx` only.
- Page break visualization or multi-page pagination in the preview. `docx-preview` renders as a continuous scroll.
- Print-from-browser functionality.
- Editing within the docx preview (it remains read-only; edits happen in the markdown textarea).
- Replacing the server-side download route with client-side generation. The download route is preserved as-is for reliability and for cases where the client-side generation might differ.
- Font embedding or Calibri font loading. `docx-preview` uses available system fonts or falls back gracefully.
- Zoom controls or page-width scaling of the preview.

## 5. Technical Notes

| Area | Detail |
|---|---|
| **New dependency** | `docx-preview` (npm: ~50KB gzipped). Renders OOXML `.docx` files into a DOM container. MIT licensed. |
| **New files** | `src/components/resume/docx-preview.tsx` — accepts `markdown: string` and `className?: string`, generates blob client-side, renders via `renderAsync()`. Wrapped in `next/dynamic` at the call site. `src/lib/docx-generator.client.ts` — imports `buildDocument()` from `docx-generator.ts`, calls `Packer.toBlob()`, exports `markdownToDocxBlob()`. |
| **Refactored file** | `src/lib/docx-generator.ts` — extract document construction into `buildDocument(markdown): Document` (shared). `markdownToDocx()` calls `buildDocument()` then `Packer.toBuffer()` (server-only, unchanged behavior). |
| **Modified files** | `src/components/resume/wizard/step-review.tsx` (swap ReactMarkdown for dynamic DocxPreview), `src/components/resume/resume-editor.tsx` (swap ReactMarkdown for dynamic DocxPreview) |
| **docx-preview API** | `renderAsync(blob, containerElement, styleElement?, options?)` — renders into a provided DOM element. Pass a `styleElement` ref to scope styles per instance. Key options: `className` for wrapper, `inWrapper: true` for page-like container, `ignoreHeight: true` for continuous scroll. |
| **Debounce** | 300ms debounce on markdown changes before re-generating. No debounce on initial mount. |
| **SSR safety** | `DocxPreview` loaded via `next/dynamic` at the call site. Inside the component, `docx-preview` is imported lazily via `import()` inside `useEffect`. `docx-generator.client.ts` is only imported by client components. |

### Component sketch

```tsx
// src/components/resume/docx-preview.tsx
"use client";

import { useEffect, useRef, useState } from "react";
import { markdownToDocxBlob } from "@/lib/docx-generator.client";
import { Loader2 } from "lucide-react";
import { cn } from "@/lib/utils";
import ReactMarkdown from "react-markdown";

interface DocxPreviewProps {
  markdown: string;
  className?: string;
}

export function DocxPreview({ markdown, className }: DocxPreviewProps) {
  const containerRef = useRef<HTMLDivElement>(null);
  const styleRef = useRef<HTMLStyleElement | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(false);
  const isInitialRender = useRef(true);

  useEffect(() => {
    if (!markdown.trim()) return;

    let cancelled = false;
    const delay = isInitialRender.current ? 0 : 300;
    isInitialRender.current = false;

    const timeout = setTimeout(async () => {
      if (!containerRef.current) return;
      setLoading(true);
      setError(false);
      try {
        const blob = await markdownToDocxBlob(markdown);
        const { renderAsync } = await import("docx-preview");
        if (cancelled) return;
        // Clear previous render
        containerRef.current.innerHTML = "";
        // Create scoped style element
        if (!styleRef.current) {
          styleRef.current = document.createElement("style");
          containerRef.current.appendChild(styleRef.current);
        }
        await renderAsync(blob, containerRef.current, styleRef.current, {
          className: "docx-preview",
          inWrapper: true,
          ignoreHeight: true,
        });
      } catch (err) {
        console.error("Failed to render docx preview:", err);
        if (!cancelled) setError(true);
      } finally {
        if (!cancelled) setLoading(false);
      }
    }, delay);

    return () => {
      cancelled = true;
      clearTimeout(timeout);
    };
  }, [markdown]);

  if (!markdown.trim()) {
    return (
      <div className={cn("rounded-md border bg-white p-4 text-sm text-muted-foreground", className)}>
        No content to preview
      </div>
    );
  }

  // Fallback to ReactMarkdown if docx rendering fails
  if (error) {
    return (
      <div className={cn("prose prose-sm max-w-none rounded-md border p-4 text-sm", className)}>
        <ReactMarkdown>{markdown}</ReactMarkdown>
      </div>
    );
  }

  return (
    <div className={cn("relative rounded-md border bg-white overflow-y-auto", className)}>
      {loading && (
        <div className="absolute inset-0 flex items-center justify-center bg-white/80 z-10">
          <Loader2 className="h-5 w-5 animate-spin text-muted-foreground" />
        </div>
      )}
      <div ref={containerRef} />
    </div>
  );
}
```

```tsx
// Call-site usage (e.g., resume-editor.tsx)
import dynamic from "next/dynamic";
const DocxPreview = dynamic(
  () => import("@/components/resume/docx-preview").then(m => ({ default: m.DocxPreview })),
  { loading: () => <div className="h-64 animate-pulse bg-muted rounded-md" /> }
);
```

## Resolved Design Decisions

| Decision | Resolution | Rationale |
|---|---|---|
| Module boundary | Shared `buildDocument()` in `docx-generator.ts` returns `Document` object. Server wrapper calls `Packer.toBuffer()`, new client file (`docx-generator.client.ts`) calls `Packer.toBlob()`. | Zero code duplication, clean client/server separation, no SSR/vitest issues. |
| Wizard preview source | Always generate client-side from markdown. | Simpler architecture, works identically for fresh generations and edited markdown. Avoids network round-trip for preview. |
| Code splitting | `DocxPreview` wrapped in `next/dynamic` with loading spinner. | Keeps ~100KB+ of docx/docx-preview libraries out of the initial bundle for users who never view a resume. |
| Debounce timing | 300ms on re-renders after markdown changes. No debounce on initial mount. | Fast enough to feel responsive, slow enough to avoid thrashing during rapid edits. Initial render should show immediately. |
| Container height | `DocxPreview` accepts a `className` prop. Callers control height. Wizard uses existing `overflow-y-auto` parent; drawer uses border container. | Each context has different layout constraints. |
| DOM cleanup | Clear `containerRef.current.innerHTML` before each `renderAsync()` call. | Prevents content stacking on re-renders. |
| Error state | Show inline error message with fallback to `ReactMarkdown` if docx rendering fails. | Graceful degradation over blank state. |
| Empty markdown | Show placeholder text ("No content to preview") for empty/whitespace-only markdown. | Avoids rendering an empty white box. |
| Dark mode | White background with border in both themes. `docx-preview` styles scoped via `styleElement` ref. | Document preview should always look like a document. Multiple instances cannot conflict. |
| Spike first | Phase 0 spike to verify `docx-preview` rendering fidelity before full integration. | If navy colors, borders, or centered text render poorly, pivot to styled ReactMarkdown (Option A) before investing in full integration. |
| Wizard height | Fixed height with internal scroll (`max-h-[70vh]` preserved). | Consistent with current layout. Avoids wizard dialog becoming excessively tall. |

# PRD: Guided Resume Generation Wizard

**Version:** 1.0
**Date:** 2026-03-03
**Author:** Product Management
**Status:** Approved
**Project:** Job Seeker -- Job Application Tracker

---

## 1. Introduction / Overview

The current resume generation flow is a single-click operation: press "Generate Tailored Resume," wait ~20 seconds, receive output. The system prompt (`resume-prompt.ts`, 38 lines) provides generic resume-writing instructions that diverge from the battle-tested `/resume` skill methodology in several critical ways:

- **Bullet philosophy**: The app says "start every bullet with a strong action verb" -- the exact pattern the skill labels "activity-first (avoid)." The skill's methodology opens bullets with business outcomes, then uses activity as evidence.
- **No fit analysis**: The skill maps experience to JD requirements before writing, identifying gaps and framing decisions. The app skips this entirely.
- **No user input**: The skill asks targeted pre-draft questions about experience gaps, title choices, and ambiguities. The app offers zero user interaction.
- **No review loop**: The skill runs a dedicated review agent that grades output (A/B/C), checks keyword alignment, narrative coherence, and company culture fit. The app does single-pass generation.
- **Missing structural guidance**: No bullet-count-per-role rules, no role consolidation logic, no title selection strategy, no 3-4 sentence summary structure.

This PRD replaces the one-click flow with a multi-step guided wizard modal. The wizard walks the user through the skill's methodology: see the fit analysis, answer targeted questions, review AI output with quality grades, and optionally iterate -- all within a modal dialog, no page navigation required.

---

## 2. Goals

- **Impact-first output quality:** Generated bullets shall lead with business outcomes, not action verbs. Output quality shall match the `/resume` skill's methodology.
- **User-guided tailoring:** Users shall answer targeted questions about experience gaps and title choices before generation, producing resumes that reflect context only the user can provide.
- **Built-in review loop:** Every generation shall include an automated quality review with keyword alignment, narrative coherence, and a letter grade. Users shall have the option to iterate.
- **Preserve existing infrastructure:** Cap management, token tracking, cost estimation, DOCX download, and generation history shall continue working unchanged.

### What Success Looks Like

A user opens an application card and clicks "Generate Resume." A modal wizard opens. Step 1 shows a fit analysis: which experiences map to JD requirements, where the gaps are. Step 2 asks 3-5 targeted questions about those gaps ("Do you have variable compensation experience?" / "Should we title this role 'Director' or 'VP'?"). Step 3 shows a loading state, then the generated resume with a quality review panel alongside: keyword alignment score, narrative assessment, letter grade, and specific bullet improvement suggestions. The user can accept, make edits, or click "Revise" to re-generate incorporating review feedback. The final resume uses impact-first bullets, has per-role bullet counts matching the skill's guidelines, and reads as a coherent career narrative -- not a keyword-stuffed list.

## 3. User Stories

### US-1: View Fit Analysis

**As a** user, **I want to** see how my experience maps to the job description before the resume is generated, **so that** I can understand which parts of my background are most relevant and where gaps exist.

**Acceptance Criteria:**
- [ ] Step 1 of the wizard shows a structured fit analysis generated by the AI
- [ ] The analysis identifies: most relevant roles, key wins that align with JD priorities, demonstrated skills, and gaps
- [ ] The analysis is presented in a scannable format (grouped sections, not a wall of text)
- [ ] The user can proceed to Step 2 or cancel the wizard

### US-2: Answer Pre-Draft Questions

**As a** user, **I want to** answer targeted questions about experience gaps and framing decisions, **so that** the generated resume reflects context that only I can provide.

**Acceptance Criteria:**
- [ ] Step 2 presents questions generated from the fit analysis in Step 1
- [ ] Questions focus on: experience gaps the JD requires, title/framing decisions, overlapping dates or ambiguities, and critical context
- [ ] Questions are presented one at a time with a "Next Question" button to advance
- [ ] Each question has a text input for free-form answers and/or selectable options where appropriate
- [ ] The user can skip individual questions via a "Skip" button (skipped questions are excluded from the generation prompt)
- [ ] The user can go back to Step 1 to review the fit analysis
- [ ] If the AI generates zero questions (no gaps found), Step 2 is skipped entirely -- the wizard jumps from Step 1 to Step 3
- [ ] Answering all questions (or skipping them) advances to Step 3

### US-3: Generate with Skill-Quality Prompting

**As a** user, **I want** the AI to use impact-first bullet methodology, structured bullet counts, and title selection logic, **so that** the output is meaningfully better than a generic resume generator.

**Acceptance Criteria:**
- [ ] The system prompt encodes the `/resume` skill's bullet-writing pattern: `[Business outcome] + [by/through] + [what you did] + [scale/evidence]`
- [ ] The prompt includes bullet-count-per-role guidance (3-4 for most recent/relevant, 2-3 for second tier, 1-2 for older, consolidate 10+ year roles)
- [ ] The prompt includes title selection logic: pick the title that best mirrors the target posting's language
- [ ] The prompt includes role consolidation guidance for multiple roles at the same company
- [ ] The professional summary follows the 3-4 sentence structure: positioning statement, proof points with numbers, credentials/breadth
- [ ] User answers from Step 2 are injected into the prompt as additional context
- [ ] The fit analysis from Step 1 is included in the prompt to guide relevance filtering

### US-4: View Quality Review

**As a** user, **I want to** see an automated quality review of the generated resume alongside the output, **so that** I can understand its strengths and weaknesses before downloading.

**Acceptance Criteria:**
- [ ] After generation completes, Step 3 shows the resume preview and a review panel
- [ ] The review is generated by a second AI call that evaluates the resume against the full JD
- [ ] The review includes: keyword alignment assessment, narrative coherence assessment, specific bullet improvement suggestions (with concrete rewrites), gaps/risks a hiring manager would flag, and an overall letter grade (A/B/C) with justification
- [ ] The review is collapsible/expandable so the user can focus on the resume preview
- [ ] The review is stored with the generation record for later reference

### US-5: Iterate with Revision Loop

**As a** user, **I want to** request a revision that incorporates review feedback, **so that** I can improve the resume without starting over.

**Acceptance Criteria:**
- [ ] A "Revise" button in Step 3 triggers a re-generation
- [ ] The revision prompt includes: the original resume, the review feedback, and any additional user notes entered in a text field
- [ ] Revision counts against the user's monthly generation cap (same as a fresh generation)
- [ ] After revision, the review endpoint is called automatically and the review panel updates with a new assessment
- [ ] The revision prompt includes the fit analysis and user answers from Steps 1-2 (same context as the initial generation), plus the review feedback and user notes
- [ ] The user can revise multiple times (cap permitting)
- [ ] The user can accept the current version at any point via "Use This Resume"

### US-6: Wizard Modal UX

**As a** user, **I want** the entire guided flow to happen in a modal without leaving the application drawer, **so that** I maintain context on the job I'm applying to.

**Acceptance Criteria:**
- [ ] The wizard opens as a full-width dialog modal (not a sheet/drawer) overlaying the application drawer
- [ ] The modal has a step indicator showing progress (Step 1/2/3)
- [ ] Navigation: "Back" and "Next"/"Continue" buttons. "Cancel" closes the modal at any step
- [ ] The modal is scrollable if content exceeds viewport height
- [ ] Closing the modal mid-flow discards wizard progress (no partial saves). If generation already completed server-side, the record persists in generation history and can be recovered.
- [ ] No confirmation dialog on close -- the user can always find completed generations in history
- [ ] On completion ("Use This Resume"), the modal closes and the generated resume appears in the application drawer's resume section as it does today

## 4. Functional Requirements

### Prompt Engineering

- **FR-1:** The system prompt shall be restructured to encode the `/resume` skill's impact-first methodology. The new prompt shall replace the current `buildResumePrompt` function in `resume-prompt.ts` (38 lines).

- **FR-2:** The system prompt shall include both positive and negative instructions for bullet-writing: (1) positive: open with business outcome / "so what," follow with the activity as evidence, and include scale/quantification; (2) negative: "DO NOT start bullets with action verbs. DO NOT use activity-first phrasing like 'Led...', 'Managed...', 'Developed...' as the opening." Both constraints prevent LLM reversion to common resume patterns.

- **FR-3:** The system prompt shall include bullet-count-per-role rules:
  - Most recent / most relevant role: 3-4 bullets
  - Second tier roles: 2-3 bullets
  - Older or less relevant roles: 1-2 bullets
  - Roles 10+ years old: consolidate into a single entry or omit

- **FR-4:** The system prompt shall instruct title selection: when the source includes alternate titles, pick the one that best mirrors the target posting's language.

- **FR-5:** The system prompt shall instruct role consolidation: multiple roles at the same company may be listed separately or consolidated depending on which tells a better narrative for the target role.

- **FR-6:** The professional summary instruction shall specify 3-4 sentences with structure: (1) positioning statement with core value proposition, (2) 1-2 proof points with specific numbers, (3) credentials and breadth of capability.

- **FR-7:** The system prompt shall instruct keyword integration to "surface phrases naturally" within bullet narratives rather than mechanically mirroring exact terminology.

### Fit Analysis (Step 1)

- **FR-8:** A new API endpoint `POST /api/resume/analyze-fit` shall accept `{ jobApplicationId }` and return a structured fit analysis as JSON:

```typescript
{
  relevantRoles: Array<{
    title: string;
    company: string;
    relevanceReason: string;
  }>;
  alignedWins: Array<{
    win: string;
    jdRequirement: string;
  }>;
  demonstratedSkills: string[];
  gaps: Array<{
    jdRequirement: string;
    assessment: string; // "missing" | "partial" | "adjacent"
    suggestion: string;
  }>;
  titleRecommendations: Array<{
    role: string;
    options: string[];
    recommendation: string;
  }>;
}
```

- **FR-9:** The fit analysis endpoint shall use Anthropic's `tool_use` (function calling) to guarantee structured JSON output. The tool schema shall match the response types defined in FR-8 and FR-11. This eliminates JSON parsing failures entirely.

- **FR-10:** The fit analysis shall be displayed in the wizard's Step 1 with clear sections: "Relevant Experience," "Aligned Wins," "Skills Match," "Gaps to Address," and "Title Options."

- **FR-10a:** The fit analysis and pre-draft questions shall be generated in a single API call. The tool schema includes both the analysis object and the questions array as top-level fields.

- **FR-10b:** Fit analysis results shall be cached per application. When the wizard opens, the endpoint checks for an existing cached analysis. The cache is invalidated when the user's resume source or the application's job description is modified. If a valid cache exists, the endpoint returns it immediately without an API call.

### Pre-Draft Questions (Step 2)

- **FR-11:** The fit analysis response shall include a `questions` array generated by the AI (within the same tool_use call as the analysis):

```typescript
questions: Array<{
  id: string;
  topic: string; // e.g., "Experience Gap", "Title Decision", "Ambiguity"
  question: string;
  type: "text" | "select";
  options?: string[]; // for select-type questions
}>
```

- **FR-12:** Questions shall be presented sequentially in the modal. Each question shows the topic label, question text, and an input field (textarea for "text" type, radio buttons for "select" type).

- **FR-13:** The user can skip any question. Skipped questions are excluded from the generation prompt context.

### Generation (Step 3 - Generation Phase)

- **FR-14:** The generation endpoint `POST /api/resume/generate` shall be extended to accept optional fields:

```typescript
{
  jobApplicationId: string;
  fitAnalysis?: object;    // structured analysis from FR-8
  userAnswers?: Array<{    // answers from Step 2
    questionId: string;
    question: string;
    answer: string;
  }>;
  revisionContext?: {      // for revision loop
    parentGenerationId: string;
    previousMarkdown: string;
    reviewFeedback: string;
    userNotes?: string;
  };
}
```

- **FR-15:** When `fitAnalysis` and `userAnswers` are provided, the user prompt shall include them as structured context sections that the model uses to inform its output.

- **FR-16:** When `revisionContext` is provided, the system prompt shall instruct the model to revise the previous resume incorporating the review feedback and user notes.

### Quality Review (Step 3 - Review Phase)

- **FR-17:** A new API endpoint `POST /api/resume/review` shall accept `{ generationId }` and return a structured review:

```typescript
{
  keywordAlignment: {
    matched: string[];
    missing: string[];
    score: string; // e.g., "8/12 key terms"
  };
  narrativeCoherence: string; // paragraph assessment
  bulletImprovements: Array<{
    original: string;
    suggested: string;
    rationale: string;
  }>;
  gapsAndRisks: string[];
  overallGrade: "A" | "B" | "C";
  gradeJustification: string;
}
```

- **FR-18:** The review endpoint shall use Anthropic's `tool_use` to guarantee structured JSON output matching the schema in FR-17. The prompt shall include: the full generated resume, the full job description, and instructions to evaluate keyword alignment, narrative coherence, bullet quality, and gaps. It shall NOT count against the generation cap.

- **FR-19:** The review result shall be persisted on the `ResumeGeneration` record (new `reviewJson` column).

### Data Model Changes

- **FR-20:** Add the following columns to `ResumeGeneration`:

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `reviewJson` | `Json?` | No | null | Structured review output from FR-17 |

| `userAnswersJson` | `Json?` | No | null | User's answers to pre-draft questions (duplicated on each record including revisions) |
| `parentGenerationId` | `String?` | No | null | FK to ResumeGeneration. Non-null indicates this is a revision of the parent generation. Enables revision rate metrics. |

- **FR-20a:** Add a `FitAnalysisCache` model for caching fit analysis results:

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `id` | String | Yes | cuid() | Primary key |
| `userId` | String | Yes | -- | FK to User |
| `jobApplicationId` | String | Yes | -- | FK to JobApplication (unique per application) |
| `analysisJson` | Json | Yes | -- | Cached fit analysis + questions |

The cache is valid when both `resumeSourceHash` and `jobDescriptionHash` match the current values. On mismatch, the cached record is replaced with a fresh analysis.

### Wizard UI

- **FR-21:** A new component `ResumeWizardModal` shall implement the multi-step dialog. It shall use the existing `Dialog` component from `src/components/ui/dialog.tsx` (158 lines).

- **FR-22:** The modal shall have 3 steps with a visual step indicator (numbered circles or progress bar):
  - Step 1: "Fit Analysis" -- displays analysis, "Continue" to proceed
  - Step 2: "Tailoring Questions" -- sequential question UI, "Generate" to proceed
  - Step 3: "Review & Finalize" -- resume preview + review panel, "Use This Resume" or "Revise"

- **FR-23:** Step 3 shall use a two-panel layout: resume preview (left/top, larger) and review panel (right/bottom, collapsible). On narrow viewports, the panels shall stack vertically.

- **FR-24:** The existing `GenerateButton` (`generate-button.tsx`, 110 lines) shall be modified to open the wizard modal instead of directly calling the generate API. The wizard is the only generation path -- there is no "quick generate" bypass.

- **FR-24a:** The wizard shall be blocked entirely when the user has reached their monthly generation cap. The "Generate Tailored Resume" button remains disabled with the existing cap-reached messaging. Users cannot enter the wizard just for fit analysis when at cap.

- **FR-25:** The fit analysis and review API calls shall track tokens and cost using the existing `estimateCost` function but shall NOT increment the user's `resumeGenerationsUsedThisMonth`. Only the actual resume generation (and revisions) shall count against the cap.

- **FR-26:** Token usage from fit analysis and review calls shall be stored in a new `ResumeAuxCall` model. Admin visibility is database-only for now; a dashboard UI can be added later if needed. Schema:

| Field | Type | Required | Default | Notes |

| `userId` | String | Yes | -- | FK to User |
| `jobApplicationId` | String | Yes | -- | FK to JobApplication |

| `modelId` | String | Yes | -- | Model used |
| `createdAt` | DateTime | Yes | now() | Timestamp |

## 5. Non-Goals (Out of Scope)

- **Streaming responses:** The wizard uses standard request/response. Streaming the resume as it generates is a future enhancement.
- **Persisting wizard state:** If the user closes the modal mid-flow, progress is discarded. No draft saving.
- **Company culture research:** The skill's Step 5 includes web-searching for company culture/leadership principles. This is excluded from the app -- the review focuses on resume content quality only.
- **DOCX generation changes:** The download flow remains unchanged. This PRD only changes how the markdown is produced.
- **Resume source changes:** The compiler (`resume-compiler.ts`, 244 lines) and source data model are unchanged.
- **Cap amount changes:** The monthly cap value and reset mechanism are unchanged.
- **Multiple resume source files:** The app continues to support one resume source per user.
- **Automatic revision:** The revision loop is user-initiated. The system does not auto-revise based on a low grade.

### User Experience

**Journey: First Generation**
1. User clicks "Generate Tailored Resume" in the application drawer
2. Wizard modal opens. Loading spinner while fit analysis runs (~5-10s)
3. Step 1: Fit analysis displayed. User reviews relevant experience, gaps, title options
4. User clicks "Continue"
5. Step 2: First question appears. User types answer or selects option, clicks "Next Question"
6. After last question, "Generate Resume" button appears
7. User clicks "Generate Resume"
8. Step 3: Loading state (~15-25s) with "Generating your tailored resume..." message
9. Resume preview appears on left. Review panel loads (~5-10s additional) on right
10. User reviews grade, reads feedback, decides to accept or revise

**Journey: Revision**
1. From Step 3, user clicks "Revise"
2. A text field appears: "Any additional guidance for the revision?" (optional)
3. User types notes or leaves blank, clicks "Revise Resume"
4. Loading state. New resume appears with updated review
5. User accepts via "Use This Resume"

**Loading States:**
- Fit analysis: "Analyzing your experience against the job description..."
- Generation: "Generating your tailored resume..." with elapsed time counter
- Review: "Running quality review..." (appears after resume is shown)
- Revision: "Revising your resume..."

**Error States:**
- API failure during fit analysis: "Failed to analyze fit. Try again?" with retry button
- API failure during generation: "Generation failed. Try again?" (cap slot is rolled back server-side via existing `rollbackGeneration()`)
- Network-level failure during generation (browser closed/timeout): generation may complete server-side, cap is consumed, result appears in generation history. No rollback for network failures -- this matches current behavior.
- API failure during review: Resume is still shown; review panel shows "Review unavailable" with retry option. User can still accept the resume without a review.

### Accessibility

- Focus shall move to the step content area when navigating between steps (managed via Radix Dialog's built-in focus trapping)
- Loading states shall use `aria-live="polite"` regions so screen readers announce progress
- Step indicator shall use `aria-current="step"` for the active step
- Letter grades in the review panel shall not rely on color alone -- the letter (A/B/C) is the primary indicator
- All buttons shall have descriptive `aria-label` attributes (e.g., "Skip this question", "Go back to fit analysis")
- Keyboard navigation: Tab through interactive elements, Escape to close modal (standard Radix behavior)

## 8. Security and Privacy

### Authentication & Authorization

All three endpoints (`analyze-fit`, `generate`, `review`) shall use the existing `authenticatedHandler` wrapper, requiring a valid session. Ownership checks verify the `jobApplicationId` belongs to the requesting user.

### Input Validation

- `jobApplicationId` validated as non-empty string via Zod schema
- `fitAnalysis` and `userAnswers` are optional JSON objects; validated for shape before inclusion in prompts
- User free-text answers are included in prompts as-is (no injection risk since they become Claude input, not executed code)
- `revisionContext.userNotes` capped at 2,000 characters

### Sensitive Data

- Resume content and job descriptions are sent to the Anthropic API (same as current behavior, no change in data exposure)
- Review feedback stored as JSON in the database; no additional PII beyond what's already in the resume

## 9. Testing Strategy

### Unit Tests

**Backend (vitest):**
- Fit analysis prompt builder produces valid system+user prompt
- Enhanced generation prompt includes impact-first instructions, bullet counts, title selection
- Generation prompt correctly injects fitAnalysis and userAnswers when provided
- Revision prompt includes previous markdown and review feedback
- Review prompt includes full resume and JD
- Token tracking for aux calls does not increment generation cap

**Frontend (vitest):**
- Wizard modal renders all 3 steps
- Step navigation: forward, backward, cancel
- Step 2 renders questions from API response and collects answers
- Step 3 displays resume preview and review panel
- "Revise" button shows text input and triggers re-generation
- "Use This Resume" closes modal and passes markdown to parent

### Integration Tests

- Full flow: analyze-fit -> generate (with answers) -> review -> revision
- Cap enforcement: generation counts, fit analysis and review do not
- Error recovery: API failure during review still allows accepting resume
- Ownership check prevents cross-user access on all three endpoints

### Edge Cases

- User with no resume source: wizard should not open (existing button disable logic)
- User at cap: wizard is blocked entirely (button remains disabled with existing messaging)
- Empty answers array: generation proceeds without user context
- Review API failure after successful generation: resume is usable, review panel shows error state
- User closes modal during generation API call: generation still completes server-side, cap is consumed, result appears in generation history for recovery
- Zero questions from fit analysis: Step 2 is skipped, wizard jumps from Step 1 directly to generation

**No new libraries required.** The feature uses:
- `@anthropic-ai/sdk` (existing) -- Claude API calls
- `@radix-ui/react-dialog` (existing) -- Modal dialog
- `react-markdown` (existing) -- Resume preview
- `zod` (existing) -- Request validation
- `prisma` (existing) -- Database operations

### Assumptions

- The current Anthropic API response time (~15-25s for generation) is acceptable for a wizard flow where the user has already spent time on Steps 1-2
- Fit analysis + review calls add ~$0.005-0.01 per generation in API cost (acceptable given they improve output quality)
- Users will engage with the wizard steps rather than wanting to skip directly to generation. The wizard is the only generation path; a "quick generate" bypass can be added later if usage data shows demand
- Anthropic's `tool_use` reliably produces structured JSON for fit analysis and review outputs, eliminating the need for JSON parsing fallbacks

### Known Constraints

- No streaming: users see loading spinners during each AI call
- Each revision consumes a generation credit, which may feel expensive to users at the default 5/month cap
- The wizard adds ~30-60s of user interaction time on top of generation time, making the total flow ~2-3 minutes vs. ~20s today. This is the intended tradeoff for higher quality output.

## Resolved Decisions

The following design decisions were resolved during PRD review:

| Decision | Resolution | Rationale |
|---|---|---|
| JSON output method | `tool_use` (function calling) | Guarantees structured JSON; eliminates parsing failures |
| Fit analysis + questions | Single API call | Faster, cheaper; tool schema includes both |
| Cap behavior at limit | Block wizard entirely | Prevents frustration of completing Steps 1-2 then hitting a wall |
| Auto-review after revision | Yes, automatic | Seamless UX; user always sees updated quality assessment |
| Zero questions handling | Skip Step 2 entirely | Fastest path; no empty-state UI needed |
| Prompt negative instructions | Include explicit "DO NOT" | Prevents LLM reversion to action-verb-first patterns |
| Page length target | Fixed 1 page always | Simpler, consistent |
| Modal close on Step 3 | Silent close; recover from history | Generation is already persisted; no data loss |
| Quick generate bypass | Not included | Wizard is the only path; bypass can be added later based on usage data |
| Phase 1 prompt shipping | Ship immediately | Improves existing one-click generations as a quick win |
| Admin aux call visibility | Database only for now | Avoids UI work; admins can query directly |
| Revision data model | `parentGenerationId` on ResumeGeneration | Enables revision rate metrics; clear lineage |
| Fit analysis + answers storage | Duplicate on each ResumeGeneration record | Self-contained records; simpler queries; acceptable duplication |
| Accessibility level | Standard a11y | Focus management, aria-live, keyboard nav, no color-only indicators |
| Wizard access point | Drawer only | No standalone page; wizard is contextual to a specific job application |
| "Use This Resume" action | Populate editor (current behavior) | Resume appears in Edit/Preview tabs in the drawer; user can edit before downloading |
| Fit analysis caching | Cache until resume/JD changes | Reuse cached analysis for an application; invalidate when resume source or job description is modified |
